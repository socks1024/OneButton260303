## 1. 当前道路系统现状

### 生成方式

道路由代码程序化生成（`road_segment.gd`），无任何美术资产。所有视觉表现都是代码创建的基础几何体 + 纯色材质。

### 关键参数

| 参数 | 值 | 说明 |
|------|----|------|
| `SEGMENT_LENGTH` | 20m | 单段路长度（沿Z轴负方向） |
| `ROAD_WIDTH` | 4m | 路面宽度 |
| `ROAD_HEIGHT` | 0.5m | 路面厚度 |
| `BASE_GAP_LENGTH` | 3m | 断口最小长度 |

### 当前节点树结构

- `RoadSegment (Node3D)` → `StaticBody3D` → `MeshInstance3D`（BoxMesh）+ `CollisionShape3D`（BoxShape3D）

有断口时
有断口时，一个路段会被分为两段 road_piece（断口前 + 断口后），另外在断口前后沿各有一个红色细长条 MeshInstance3D 作为警示标记。

### 核心约束

- **碰撞体必须保持简单的 BoxShape3D**，不能替换为复杂碰撞网格
- 玩家通过 `RayCast3D`（向下探测 2m）检测前方地面是否存在，以此判断断口并自动跳跃
- 路段在玩家前方 5 段（约 100m）范围内动态生成，超过后方 2 段的路段被回收

### 3.1 路面模型替换

**目标**：将程序生成的 BoxMesh 替换为美术制作的路面 3D 模型。

#### 🎨 美术师需要做的

1. **制作一个标准路面模型**
   - 尺寸基准：**宽 4m × 长 20m × 厚 0.5m**（与代码常量匹配）
   - 原点设置在路面**顶面的起始端中心**（X=0, Y=0, Z=0 对应路面顶面起始端中心点）
   - 路面沿 **Z 轴负方向** 延伸 20m
   - 路面表面（Y=0 平面）必须保持平坦，确保碰撞体（Box）不会穿帮
   - 路面底部和侧面可以有造型细节（如悬崖、裂纹、岩石质感等）

2. **可选：制作多个变体**
   - 建议制作 3~5 个路面变体，随机使用以减少重复感
   - 每个变体尺寸和原点必须一致

3. **输出格式**
   - 导出为 `.glb` 格式
   - 存放路径建议：`Content/Art/Model/Road/road_piece_01.glb` 等
   - 面数建议：**单段 ≤ 500 面**（同屏最多约 7 段路，总计 ≤ 3500 面）

4. **有断口时的模型（Shader 裁剪方案）**
   - 断口处会把一段 20m 的路切成两截，每截长度不固定
   - **采用 Shader 裁剪方案**：美术始终提供完整 20m 模型，程序端通过自定义 Shader 的 `discard` 将断口区域像素丢弃——模型不缩放、不拆分，纹理零变形
   - 美术额外提供**断口边缘「断面」模型**（见 3.3 节），在 Shader 裁剪的基础上叠加碎裂/崩塌的视觉效果

5. **断面边缘模型（gap_edge.glb）**
   - 宽度与路面一致：**4m**
   - 深度较短：约 **0.5~1m**（只负责断裂边缘的过渡造型）
   - 高度与路面一致：**0.5m**
   - 造型：断裂、碎裂、崩塌的效果——不是平整的切面，而是参差不齐的碎裂边缘
   - 原点在**断裂面那一侧的中心**（方便程序端对齐断口边界）
   - 断裂面朝 Z 轴负方向
   - 建议制作 **2~3 个变体** 增加随机感
   - 存放路径：`Content/Art/Model/Road/gap_edge_01.glb` 等

6. **溶解噪声纹理（dissolve_noise.png）**
   - Shader 裁剪边缘会使用噪声纹理产生「腐蚀/溶解」过渡效果
   - 尺寸：256×256，灰度，四方连续（tileable）
   - 格式：`.png`
   - 存放路径：`Content/Art/Texture/Road/dissolve_noise.png`

#### 💻 程序员需要做的

##### 1) 创建路面裁剪 Shader（`road_clip.gdshader`）

- 使用 `discard` 基于世界坐标 Z 值裁剪断口区间内的像素
- 裁剪边缘使用溶解噪声纹理产生「腐蚀/溶解」过渡效果，并叠加自发光
- `clip_z_min` / `clip_z_max` 控制裁剪区间（`clip_z_min` > `clip_z_max`，如 `-5.0` 到 `-8.0`）
- 当两者相等时表示无裁剪

##### 2) 创建 `.tscn` 继承场景（编辑器中完成，无需代码）

- 在 Godot 编辑器中右键 `.glb` 文件 → **"新建继承场景"** → 保存为同名 `.tscn`（如 `road_piece_01.tscn`）
- 在继承场景中找到 `MeshInstance3D` 节点，将 **`Surface Material Override`** 设为 `road_clip.tres`（ShaderMaterial）
- 此时编辑器中即可**实时预览** Shader 溶解效果，方便美术/TA 联调
- 每个路面变体 `.glb` 都需要对应创建一个 `.tscn` 继承场景
- 美术更新模型时只需重新导出 `.glb`，继承场景中的材质覆盖和额外节点保持不变

##### 3) 修改 `road_segment.gd`

- 始终放置完整 20m 路面场景（`preload` 对应的 `.tscn`，不再加载 `.glb`），通过 Shader uniform 参数（`clip_z_min` / `clip_z_max`）控制断口区域隐藏
- 碰撞体只覆盖实际路面部分（断口前后各一个 BoxShape3D）
- 断口两端各放置一个断面边缘模型（后端旋转 180°）
- **无需运行时遍历/替换材质**——材质已在 `.tscn` 中预配置好

##### 4) Shader 溶解带与断面模型的空间衔接（⚠️ 重要）

溶解 Shader 过渡带与断面碎裂模型在断口边界处存在空间重叠，需要合理控制各自的范围以避免视觉冲突：

```
        路面完整区域        溶解过渡    断面模型      断口空白
    ████████████████████ ▓▓▓░░░ [碎裂碎裂]          (深渊)
                        ←0.5m→←0.5~1m→←  gap_length  →
                              ↑
                          断口边界(clip_z_min)
```

**三条原则：**

1. **Shader 溶解带**负责"软过渡"——像素从完整逐渐腐蚀消失（`edge_width` 建议 **0.3~0.5m**）
2. **断面模型**负责"硬造型"——碎裂的几何边缘（深度约 **0.5~1m**）
3. **`edge_width` 不要大于断面模型的深度**——确保断面模型所在区域的路面像素已被 Shader 大部分/完全 `discard`，避免两者表面重合导致 Z-fighting 闪烁

**潜在问题与解决方式：**

| 问题 | 原因 | 解决方式 |
|------|------|---------|
| Z-fighting 闪烁 | 断面模型表面与路面模型表面在相同 Y 坐标重合 | 断面模型顶面略高于路面（Y 偏移 0.01m），或断面造型本身就是下沉/碎裂的，不与路面齐平 |
| 视觉争抢/混乱 | 溶解带过宽与断面模型范围大量重叠 | 控制 `edge_width` ≤ 断面模型深度；两者应"接力"而非完全重叠 |

**实际调参建议：** 将 Shader 的 `edge_width` 暴露为可调参数，美术与程序联调时微调至最佳效果。

### 3.2 路面材质/纹理

**目标**：为路面添加贴图材质，取代当前的纯色 `StandardMaterial3D`。

#### 🎨 美术师需要做的

1. **制作路面纹理贴图**（契合噩梦/民俗恐怖主题）
   - **Albedo 贴图**（必需）：路面基础颜色，可以是破旧的石板路、泥泞的土路、裂开的水泥路等
   - **Normal 贴图**（推荐）：增加凹凸细节感
   - **Roughness 贴图**（可选）：控制反光/湿润度
   - 尺寸建议：1024×1024 或 2048×512（横向铺 4m，纵向重复平铺）
   - 格式：`.png`
   - 纹理必须**四方连续（tileable）**——路面是无限延伸的，纹理会重复平铺

2. **存放路径建议**
   - `Content/Art/Texture/Road/road_albedo.png`
   - `Content/Art/Texture/Road/road_normal.png`

#### 💻 程序员需要做的

- 如果仍使用代码生成 BoxMesh，在 `_create_road_piece()` 中将纯色材质替换为加载美术贴图的 `StandardMaterial3D`，并设置 UV 平铺
- 如果使用美术模型（3.1），`.glb` 中内嵌的材质仅用于建模软件中预览，**实际运行使用的是 `.tscn` 继承场景中覆盖的 ShaderMaterial**（见 3.1 节），程序端无需额外处理材质替换

### 3.3 断口边缘警示效果

**目标**：替换当前简陋的红色细条，改为更有氛围的警示效果。断口边缘的视觉表现现在分为**三层叠加**，可按需逐步实现。

#### 三层效果概览

| 层级 | 效果 | 来源 | 优先级 |
|------|------|------|--------|
| 第1层 | Shader 噪声溶解边缘 + 发光 | `road_clip.gdshader`（见 3.1 节） | 🔴 高（随路面 Shader 自动生效） |
| 第2层 | 断面碎裂模型 | `gap_edge_xx.glb`（见 3.1 节） | 🔴 高（随路面模型替换自动生效） |
| 第3层 | 粒子特效 / 点光源 | 额外添加 | 🟡 中（锦上添花） |

> **注意**：第 1 层和第 2 层的美术需求与程序实现已在 **3.1 节** 中详细描述（Shader 裁剪方案 + 断面模型），两层之间的空间衔接注意事项也在 3.1 节末尾说明。此处仅补充第 3 层的内容。

#### 🎨 美术师需要做的（第 3 层：粒子特效）

1. **断口边缘粒子特效**
   - 制作断口边缘的烟雾/火焰/红光粒子效果
   - 导出为 Godot 的 `GPUParticles3D` 场景（`.tscn`）
   - 或提供粒子纹理（sprite sheet），由程序端配置
   - 颜色建议与 Shader 溶解边缘发光色一致（默认暗红色 `0.8, 0.2, 0.1`）

2. **断口深渊雾气粒子**（可选）
   - 由于道路是悬浮在虚空中的桥，断口下方与玩家周围的虚空是**连通的**，无需单独制作深渊底部/侧壁
   - 但可以制作从断口处**向上飘出的暗色雾气/幽灵光点**粒子，暗示断口下方的无尽虚空
   - 颜色偏暗紫/暗红，与整体噩梦氛围统一

3. **存放路径**
   - `Content/Art/VFX/Road/gap_particles.tscn`

#### 💻 程序员需要做的（第 3 层）

在 `_build_road()` 的断口处理分支中（见 3.1 节），追加粒子特效和点光源（位置居中于断口区域）。断口处不需要额外的深渊平面或侧壁——整体虚空氛围由天空盒和雾气统一营造（见 3.6、3.7 节）。

### 3.4 道路两侧环境装饰

**目标**：在悬浮桥体周围添加环境物件，丰富空间感，强化噩梦主题。

> **场景前提**：道路是一座悬浮在虚空中的桥，两侧没有地面，因此所有装饰物必须以**桥体附属、悬浮漂浮、垂挂悬吊**的形式出现，不能使用需要地面支撑的物件（如枯树、墓碑、碎石堆等）。

#### 🎨 美术师需要做的

1. **制作装饰物件模型集**（选择符合噩梦/民俗恐怖主题 + 悬浮场景的物件）

   | 类型 | 放置方式 | 示例 | 建议面数 | 数量 |
   |------|---------|------|---------|------|
   | 桥体附属物 | 固定在桥面边缘 | 残破栏杆、断裂桥柱、锈蚀路灯、悬挂的锁链 | ≤ 200 面 | 3~5 种 |
   | 悬浮漂浮物 | 漂浮在桥两侧虚空中 | 漂浮的碎石块、悬空的纸符/经幡、飘浮的残破灯笼 | ≤ 150 面 | 4~6 种 |
   | 远景剪影 | 远处虚空中的大型轮廓 | 漂浮的巨大建筑残骸、模糊的巨型轮廓 | ≤ 300 面 | 2~3 种 |

2. **导出要求**
   - 格式：`.glb`
   - **桥体附属物**：原点在物件与桥面的连接点处（方便对齐桥面边缘）
   - **悬浮漂浮物**：原点在物件几何中心（方便程序端控制漂浮动画的旋转/摆动轴心）
   - **远景剪影**：原点在物件中心
   - 存放路径：`Content/Art/Model/Road/Deco/`

3. **注意事项**
   - 所有装饰物件**无碰撞体**，纯视觉
   - 悬浮物建议自带倾斜/不规则朝向，增强失重感
   - 部分物件可带 Emission（如幽灵灯笼的微弱光芒、纸符的暗红荧光）
   - 远景剪影使用极低面数 + 暗色半透明材质即可，不需要细节

#### 💻 程序员需要做的

在 `RoadSegment` 或新建 `RoadDecorator` 脚本中，生成路段时按类型放置装饰物：

- **桥体附属物**：X 轴放置在桥面边缘（X ≈ ±2m），Y=0（与桥面齐平），每段路 1~3 个
- **悬浮漂浮物**：X 轴放置在桥两侧虚空中（X = ±3m ~ ±8m），Y 轴随机偏移（-2m ~ +2m，高低不一），每段路 2~4 个；可添加缓慢浮动/摆动的 Tween 动画
- **远景剪影**：X 轴距离较远（X = ±15m ~ ±30m），Y 轴随机偏移（-10m ~ +5m），每 2~3 段路放置 1 个；可使用 Billboard 或极低 LOD
- 从预加载的装饰物场景数组中随机择一实例化

### 3.5 路面表面装饰物

**目标**：在路面上点缀一些小物件（不影响碰撞），打破路面的单调感。

#### 🎨 美术师需要做的

1. **制作路面小物件**
   - 类型：路面裂缝贴花、散落的纸符、脚印、水渍、落叶等
   - 可以是 3D 小模型（≤ 20 面），也可以是贴花纹理（Decal 用途）
   - 贴花纹理尺寸：256×256，带 Alpha 通道，`.png` 格式

2. **存放路径**
   - 3D 小物件：`Content/Art/Model/Road/SurfaceDeco/`
   - 贴花纹理：`Content/Art/Texture/Road/Decals/`

#### 💻 程序员需要做的

使用 Godot 的 `Decal` 节点在路面上投射贴花：

- 每段路随机放置 0~3 个贴花
- X 位置限制在路面宽度范围内并留有边距
- Decal 节点略高于路面（Y=0.1），随机旋转朝向

### 3.6 雾气/氛围粒子

**目标**：在道路区域添加弥漫的雾气或悬浮粒子，营造噩梦氛围。整体虚空/深渊的恐惧感也主要由此节的雾气 + 3.7 节的天空盒来统一营造（道路是悬浮在虚空中的桥，断口下方与周围虚空连通，无需单独做深渊效果）。

> **本节提供四种雾气实现方案，可单独使用，也可组合叠加。推荐由 TA 根据实际效果和性能预算选择最终方案。**

#### 方案总览

| 方案 | 效果质量 | 性能开销 | 局部控制 | 开发难度 | 渲染器要求 |
|------|---------|---------|---------|---------|-----------|
| A. 环境雾 | ⭐⭐ | 极低 | ❌ | 极低 | 全部支持 |
| B. 体积雾 + FogVolume | ⭐⭐⭐⭐ | 中 | ✅ | 低 | Forward+ |
| C. 粒子雾 | ⭐⭐⭐ | 中~高 | ✅ | 中 | 全部支持 |
| D. Shader 屏幕后处理雾 | ⭐⭐⭐⭐ | 低~中 | ✅ | 高 | 全部支持 |

#### 推荐组合（效果最佳）

如果项目使用 **Forward+** 渲染器，推荐三层叠加：

1. **体积雾（方案 B）** — 负责全局大氛围，远处自然消隐
2. **FogVolume（方案 B）** — 负责局部差异（桥下深渊浓雾、断口处加浓）
3. **粒子（方案 C）** — 负责动态细节点缀（桥面上少量飘浮粒子）

```
             粒子点缀（方案C）
                ·  ·  ·
         玩家      ·    ·
          ↓
═══════[桥面]═══════           ← 全局体积雾薄雾（方案B 全局）
          ║
    [断口浓雾 FogVolume]       ← 断口处单独加浓（方案B FogVolume）
          ║
████████████████████████       ← 桥下方巨大 FogVolume 深渊浓雾（方案B FogVolume）
```

---

#### 方案 A：环境雾（Environment Fog）

最简单、性能开销最低的方案。在 `WorldEnvironment` 节点的 `Environment` 资源中直接配置。

**💻 程序员 / TA 配置**

在 `game_world.tscn` 的 `WorldEnvironment > Environment` 中：

| 属性 | 说明 | 建议值 |
|------|------|--------|
| `Fog > Enabled` | 开启雾气 | `true` |
| `Fog > Light Color` | 雾的颜色 | 暗紫/深灰 `(0.05, 0.02, 0.08)` |
| `Fog > Density` | 雾的密度 | `0.02 ~ 0.05` |
| `Fog > Sky Affect` | 雾对天空盒的影响程度 | `0.5 ~ 1.0` |
| `Fog > Height` | 高度雾的基准 Y 值 | `-5`（桥面下方更浓） |
| `Fog > Height Density` | 高度雾的密度 | `0.1 ~ 0.3` |

**优点**：零性能开销、全局统一、配置简单  
**缺点**：效果均匀，无法做局部浓淡变化，视觉上比较"死板"

**🎨 美术师需要做的**：无额外资产需求。

---

#### 方案 B：体积雾 + FogVolume（Volumetric Fog）⭐ 推荐

Godot 4.x 原生支持的后处理体积雾，效果远比环境雾好，且支持用 `FogVolume` 节点在特定区域创建局部浓雾。

> ⚠️ **仅在 Forward+ 渲染器中可用**，不支持 Compatibility 渲染器。

**💻 程序员 / TA 配置**

##### 全局体积雾

在 `WorldEnvironment > Environment` 中：

| 属性 | 说明 | 建议值 |
|------|------|--------|
| `Volumetric Fog > Enabled` | 开启体积雾 | `true` |
| `Volumetric Fog > Density` | 全局密度 | `0.02 ~ 0.06` |
| `Volumetric Fog > Albedo` | 雾的颜色 | 暗紫/暗灰 |
| `Volumetric Fog > Emission` | 雾的自发光 | 可设为微弱暗红，营造地狱感 |
| `Volumetric Fog > Length` | 雾效计算距离 | `64 ~ 128` |
| `Volumetric Fog > Anisotropy` | 各向异性（光在雾中的散射方向） | `0.3`（正值 = 前向散射，光源会有"光柱"效果） |

##### 局部 FogVolume 节点

在场景树中添加 `FogVolume` 节点，可在特定区域创建局部浓雾：

```
FogVolume 节点
├── Shape: Box / Cylinder / World（投影形状）
├── Size: Vector3（覆盖范围）
└── Material: FogMaterial
    ├── Density: 局部密度
    ├── Albedo: 局部颜色
    └── Emission: 局部发光
```

**适用场景举例**：
- 在断口处放一个 `FogVolume`（Box 形状），让断口区域雾气更浓
- 在桥面下方放一个巨大的 `FogVolume`，模拟虚空深渊的浓雾
- 在桥面上方放一个薄薄的 `FogVolume`，模拟低空弥漫的雾气

**程序端**：可在 `road_segment.gd` 的断口处理分支中动态生成 `FogVolume` 节点，用代码设置其 `size` 和 `position`。

**优点**：效果真实（光线与雾交互、有明暗变化）、支持局部控制、与光源自动交互产生光柱效果  
**缺点**：有一定 GPU 开销（中低端设备需注意），仅支持 Forward+ 渲染器

**🎨 美术师需要做的**：无额外资产需求（FogMaterial 在编辑器中配置即可）。

---

#### 方案 C：粒子雾（GPUParticles3D）

用粒子系统模拟雾气，视觉上有"飘动感"，适合做动态细节点缀。

**🎨 美术师需要做的**

1. **雾气粒子纹理**
   - 制作柔和的烟雾 sprite，带 Alpha 通道
   - 尺寸：128×128 或 256×256
   - 颜色偏暗（深灰/暗紫/暗红）
   - 格式：`.png`
   - 存放路径：`Content/Art/VFX/Road/fog_particle.png`

2. **悬浮光点纹理**（可选）
   - 小型圆形光点，带发光效果
   - 尺寸：32×32
   - 颜色：幽绿/暗红
   - 存放路径：`Content/Art/VFX/Road/float_light.png`

**💻 程序员需要做的**

两种放置方式：

- **全局粒子（跟随玩家）**：在 `Player` 场景中添加 `GPUParticles3D` 作为子节点，Box 发射形状覆盖玩家周围区域，粒子随玩家移动
- **局部粒子（每段路上生成）**：在 `road_segment.gd` 中添加 `GPUParticles3D`，Box 发射形状覆盖整段路面（4m × 2m × 20m），约 20 个粒子，微弱上浮，寿命 4 秒

**优点**：有动态飘动感、全部渲染器均支持、可做局部效果  
**缺点**：大面积覆盖不如体积雾自然，粒子数多了性能下降

---

#### 方案 D：Shader 屏幕后处理雾

通过自定义屏幕空间 Shader 实现，挂在全屏 `ColorRect` 上做后处理，可完全自定义雾的算法。

**💻 程序员 / TA 需要做的**

1. 创建全屏 `ColorRect`，挂载自定义 `ShaderMaterial`
2. 在 Shader 中读取深度缓冲（`hint_depth_texture`），根据深度计算雾的浓度
3. 可自定义各种非写实风格化效果（径向雾、脉动雾、颜色渐变等）

**优点**：完全可控，能做各种非写实风格化雾效，全部渲染器均支持  
**缺点**：需要编写 Shader，开发成本高；不与光源交互（无光柱效果）

**🎨 美术师需要做的**：无额外资产需求（视具体 Shader 设计可能需要噪声纹理）。

### 3.7 光照与阴影

**目标**：为道路场景配置合适的光照方案，增强画面深度和恐怖氛围。

#### 🎨 美术师需要做的

1. **提供天空盒 / HDRI**（可选）
   - 噩梦主题的天空（血红色、暗紫色、阴云密布等）
   - 格式：`.hdr` 或 `.exr`
   - 存放路径：`Content/Art/Texture/Sky/`

2. **提供光照参考**
   - 整体偏暗，玩家前方有微弱的照明
   - 色调建议：冷色调（偏蓝/偏紫），断口附近可以有暖色点光源（红/橙）

#### 💻 程序员需要做的

在 `game_world.tscn` 中配置：

1. **方向光（模拟月光/噩梦之光）**
   - Color: 偏蓝冷色 (0.6, 0.65, 0.8)，Energy: 0.3~0.5（偏暗），开启阴影，从上方斜射

2. **玩家头顶跟随点光源**（在 `Player` 场景中添加 `OmniLight3D`）
   - Color: 暖白色，Energy: 0.5~0.8，Range: 10~15m，平方衰减，开启阴影

3. **断口处可选添加点光源**（已在 3.3 节断口边缘警示效果中包含，见其第 3 层实现代码）

### 3.8 路段接缝过渡

**目标**：消除相邻路段之间的视觉接缝，使无限路面看起来是连续的。

#### 🎨 美术师需要做的

1. **路面模型的 Z 轴两端必须完全平齐**（Y=0 平面在 Z=0 和 Z=-20m 处的截面完全吻合）
2. **纹理在 Z 方向上必须四方连续**（tileable），确保两段路拼接时纹理无缝
3. **如果路面有侧面造型**（如悬崖壁），在 Z=0 和 Z=-20m 的截面处保持简单的平面切口

#### 💻 程序员需要做的

1. 确保 `_next_segment_z` 的计算精确，每段路之间无间隙（`road_manager.gd` 中已有 `_next_segment_z -= RoadSegment.SEGMENT_LENGTH`，无需修改）
2. 如果使用纹理平铺，注意 UV 偏移量跨段连续（可通过将全局 Z 坐标传入材质的 `uv1_offset` 实现）

## 4. 通用约束与注意事项

### 对美术师

| 约束项 | 要求 |
|--------|------|
| 坐标系 | Godot 使用 Y-up 右手坐标系，路面在 XZ 平面上，Y=0 为路面顶部 |
| 前进方向 | Z 轴**负方向**为玩家前进方向 |
| 模型原点 | 路面模型原点在**顶面起始端中心**（即 X=0, Y=0, Z=0 处为路段开头的顶面中心） |
| 面数预算 | 路面单段 ≤ 500 面，装饰物单个 ≤ 300 面，同屏总面数建议 ≤ 10000 面 |
| 导出格式 | 统一使用 `.glb`（Godot 原生支持最好） |
| 纹理格式 | `.png` |
| 纹理尺寸 | 不超过 2048×2048，路面纹理建议 1024×1024 |
| 命名规范 | 小写下划线命名，如 `road_piece_01.glb`、`road_albedo.png` |

### 对程序员

| 约束项 | 要求 |
|--------|------|
| 碰撞体 | 始终使用简单的 `BoxShape3D`，不随美术模型改变 |
| 射线兼容 | 玩家的 `RayCast3D` 从 Y=0.2 向下探测 2m，碰撞层必须保持一致 |
| 对象池 | 路段被 `queue_free()` 回收，装饰物/粒子作为子节点会自动释放 |
| 性能 | 注意光源数量（建议同屏 ≤ 3 个动态光源），粒子数量（单段 ≤ 50 个粒子） |
| 资源加载 | 使用 `preload()` 预加载美术资源，避免运行时卡顿 |
| 模型变体 | 随机选择路面变体时使用 `Array.pick_random()`，确保视觉多样性 |

## 5. Godot 标准 3D 美术资产工作流参考

> 本节介绍 Godot 中 3D 美术资产从建模到引擎内使用的标准工作流，供美术师和程序员对齐流程认知。

### 5.1 资产导入流程

```
建模软件 (Blender / Maya / 3ds Max)
    │
    ▼ 导出 .glb / .gltf
    │
Godot 自动导入 → 生成 .glb.import 缓存
    │
    ▼ 右键 .glb → "新建继承场景"
    │
.tscn 继承场景（可在编辑器中自由修改材质、添加节点等）
    │
    ▼ 代码中 preload / load 该 .tscn
    │
运行时实例化使用
```

### 5.2 `.glb` vs `.tscn` 继承场景

| 项目 | 直接使用 `.glb` | 使用 `.tscn` 继承场景 |
|------|-----------------|----------------------|
| 修改材质 | ❌ 需要在导入设置或代码中处理 | ✅ 直接在 Inspector 中覆盖 |
| 添加子节点 | ❌ 无法直接添加 | ✅ 自由添加（粒子、光源、碰撞体等） |
| 编辑器预览 | 仅原始材质 | ✅ 可预览自定义 Shader 效果 |
| 美术更新模型 | 重新导出 `.glb` 即可自动更新 | ✅ `.glb` 更新后继承场景自动继承变更 |
| 适用场景 | 无需任何修改的简单模型 | **需要自定义材质/添加节点的模型（推荐）** |

### 5.3 材质覆盖机制

Godot 的 `MeshInstance3D` 提供三级材质优先级（从高到低）：

1. **`Material Override`**（材质覆盖）— 强制所有表面使用同一材质
2. **`Surface Material Override [n]`**（表面材质覆盖）— 按表面索引单独覆盖
3. **`Mesh` 内嵌材质** — `.glb` 导入时自带的材质（最低优先级）

在 `.tscn` 继承场景中设置 `Surface Material Override` 即可用自定义 `ShaderMaterial` 覆盖 `.glb` 内嵌材质，而不影响原始 `.glb` 文件。

### 5.4 美术迭代标准流程

```
美术师修改模型 → 重新导出覆盖 .glb → Godot 自动重新导入
                                          │
                                          ▼
                                .tscn 继承场景自动继承几何变更
                                （材质覆盖、额外子节点等保持不变）
                                          │
                                          ▼
                                编辑器中实时看到更新效果
```

> **关键**：美术师只需关心 `.glb` 的导出，程序/TA 在 `.tscn` 中配置材质和节点。两者互不影响，可并行工作。

### 5.5 本项目实际应用

本项目路面模型采用上述标准工作流：

1. 美术导出 `road_piece_xx.glb`（含内嵌预览材质）
2. 在 Godot 编辑器中创建 `road_piece_xx.tscn` 继承场景
3. 在继承场景中将 MeshInstance3D 的材质覆盖为 `road_clip.tres`（ShaderMaterial）
4. 程序端 `preload("road_piece_xx.tscn")` 并实例化，通过设置 Shader uniform（`clip_z_min` / `clip_z_max`）控制断口裁剪
5. 美术更新模型时只需重新导出 `.glb`，材质配置和 Shader 参数不受影响
